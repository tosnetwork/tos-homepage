One Masterchain, Three Execution Domains

Different workloads want different trade-offs. TOS runs a native sharded TVM layer, an Ethereum-compatible EVM workchain, and UNO privacy in parallel instead of forcing everything into one VM.

Native Layer (wc=0)

Asynchronous, sharded TVM execution for message-driven apps. This is the high-throughput native layer for TOS contracts, batch fan-out, and core protocol primitives.

EVM Workchain (wc=1)

A synchronous Ethereum-semantic execution domain. Deploy unmodified Solidity contracts and talk to it through standard Ethereum JSON-RPC served directly by validator-engine.

Uno Privacy (wc=2)

A PQ-native privacy workchain: shielded note-pool transfers, Plonky3 STARK proofs, Poseidon2, and hybrid ECDH + ML-KEM-768 note encryption. Bridgeless by architecture.

One validator-engine

One node binary serves all three workchains. Public APIs, validation, liteserver duties, and workchain dispatch all anchor back to the same root of truth.

One canonical CLI

`tosctl` is the operator and wallet entry point for the network: node operation, wallet flows, and workchain-specific tooling live on a single path instead of fragmented sidecars.

Unified JSON-RPC

TOS-native `tos_*` methods, standard `eth_*` on wc=1, and `uno_*` on wc=2 are all surfaced from the node. One RPC surface, three execution domains.

Cell-native state

TVM cells, EVM account state, and Uno shielded state all commit into TOS cells. The storage layer stays uniform even as execution models differ.

Deterministic execution

Consensus-critical paths avoid wall-clock reads, OS RNG, and nondeterministic map iteration. The repo treats determinism as a protocol invariant, not SDK hygiene.

Shared masterchain consensus

`wc=0`, `wc=1`, and `wc=2` are first-class workchains under one masterchain-rooted consensus. Users choose the execution domain per transaction; validators opt in per workchain.

Native ↔ EVM value flow

Cross-workchain value transfer between the native layer and `wc=1` uses the TOS message model. Uno stays isolated on purpose: no shield, no unshield, no bridge.

Catchain foundation

Block production is rooted in Catchain, with ongoing convergence toward a Simplex-oriented design for cleaner proof-of-progress and validator behavior.

Ethereum toolchain compatible

MetaMask, ethers.js, web3.py, Foundry, Hardhat, and Remix connect to `wc=1` without adapters. Existing Solidity codebases deploy as-is.

PQ-native privacy posture

Uno ships with hash-based STARK proving and hybrid PQ note encryption from day one, reducing the migration debt that pairing-based privacy systems still carry.

Plonky3 + ML-KEM

Uno combines Plonky3 STARKs over Goldilocks with Poseidon2 and ML-KEM-768 hybrid note encryption. Wallets prove locally; validators verify deterministically.

Validator + liteserver ops

The repo already carries `validator-engine`, liteserver, proxy-liteserver, and related operator tooling. TOS is built around a real node stack, not just contract deployment UX.

Source-first technical docs

`README`, `BUILD.md`, EVM topology docs, Uno design docs, Catchain papers, and changelogs in the repo are the fastest way to understand what is real, shipped, and in flight.

ADNL / DHT / RLDP / QUIC

TOS includes its own network stack: ADNL identities, DHT discovery, RLDP transport, QUIC interop tests, and validator overlays for the node-to-node path.

Sites, DNS, Connect

Beyond raw chain execution, the codebase already includes TOS Connect wallet flows, TOS DNS name resolution, and TOS Sites style decentralized serving primitives.

Frequently Asked Questions

TOS is a multichain Layer-1 with one masterchain-rooted consensus and three execution domains: asynchronous native TVM on wc=0, Ethereum-compatible EVM on wc=1, and PQ-native Uno privacy on wc=2. One validator-engine, one tosctl path, and one cell-native state model tie the stack together.

Native Layer (wc=0)

Asynchronous, sharded TVM execution gives TOS a message-driven base layer for high-throughput contracts, actor-style apps, and protocol-native primitives.

EVM Workchain (wc=1)

wc=1 preserves Ethereum semantics and standard JSON-RPC, so Solidity contracts, MetaMask, ethers.js, Foundry, Hardhat, and Remix can work without adapters.

Uno Privacy (wc=2)

Uno is a bridgeless shielded note-pool workchain built around Plonky3 STARKs, Poseidon2, and hybrid ECDH + ML-KEM-768 note encryption.

One Node, One CLI, One RPC Surface

validator-engine serves the public APIs and validation path, while tosctl stays the canonical operator and wallet entry point across the stack.

Cell-Native and Deterministic

TVM, EVM, and Uno commit into the same cell-based storage model, and consensus-critical execution avoids wall-clock time, OS RNG, and nondeterministic iteration.

wc=1 is designed to feel like Ethereum instead of an EVM-shaped wrapper. The node serves standard eth_* JSON-RPC directly, and the repo positions MetaMask, ethers.js, web3.py, Foundry, Hardhat, and Remix as first-class tooling.

Uno is not a mixer bolted onto a public asset. It is a separate native privacy workchain whose supply is born shielded at genesis, with no shield, no unshield, and no bridge to other workchains as a permanent architectural rule.

The codebase is built around validator-engine plus the surrounding node stack, with tosctl as the canonical CLI. BUILD.md also documents liteserver, proxy-liteserver, Rust-side tooling, and the extra prerequisites needed to build Uno end to end.

The repository already includes TOS Connect wallet-to-dApp flows, TOS DNS name resolution, TOS Sites style decentralized serving, and the ADNL / DHT / RLDP / QUIC networking stack that supports node and product connectivity.

TOS does not force every workload into one execution model. Native TVM, EVM, and Uno privacy stay separate where their invariants differ, but they still share one masterchain-rooted consensus, one operator path, and one state substrate.
